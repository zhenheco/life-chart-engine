# life-chart-engine

**Детерминированное нативное вычисление трёх систем жизненных карт — западная натальная астрология, 人類圖 (Human Design) и 紫微斗數 (Zi Wei Dou Shu) — из единой записи о рождении.**

[![License: AGPL-3.0](https://img.shields.io/badge/License-AGPL--3.0-blue.svg)](./LICENSE)
[![Python 3.12](https://img.shields.io/badge/Python-3.12-3776AB.svg)](#why-cpython-312-specifically)
[![No telemetry · offline](https://img.shields.io/badge/no%20telemetry-offline-green.svg)](#faq)

`life-chart-engine` — это небольшой автономный инструмент командной строки. Вы предоставляете данные о рождении одного человека — дату, время, часовой пояс и координаты места — и он вычисляет три независимые системы карт за один проход, затем выдаёт либо человекочитаемый отчёт Markdown, либо структурированный объект JSON для использования программами и агентами ИИ.

Он создан для людей, которые хотят **воспроизводимые, верифицируемые** вычисления карт вместо закрытого веб-приложения: практиков, разработчиков, создающих инструменты самопознания, и агентов ИИ, которым нужен чистый вычислительный шаг. Каждое число получено из реальных астрономических вычислений (Swiss Ephemeris) и реальной библиотеки 紫微斗數 (iztro) — не из удалённого сервиса, не из кэшированных справочников и никогда не по сети.

---

## Что вычисляет

Машина работает с тремя системами для **одного момента рождения**, поэтому их результаты могут подтверждать друг друга. Каждая система отвечает на разные вопросы:

| Система | Что это (простыми словами) | Что возвращает машина |
|---------|---------------------------|----------------------|
| **Западная натальная** (Тропическая / Placidus) | Классическая западная астрология — где планеты находились против зодиака в момент вашего рождения, разделённые на 12 домов. | Асцендент + Середина неба, 12 планет/точек (太陽 → 南交點) со знаком, градусом, домом и флагом ретроградности, все 12 куспидов домов и каждый обнаруженный аспект (合相/六分/四分/三分/對分), отсортированные по точности. |
| **人類圖 Human Design** | Современный синтез астрологии, И-Цзина и системы чакр. Описывает, как ваша энергия «подключена» через врата, каналы и центры. | Тип, Авторитет, Профиль, Определённость, Воплощённый Крест, дата Дизайна (на 88° ранее), определённые/открытые центры, определённые каналы и активации врат (для каждой планеты) для обеих карт — Личности и Дизайна. |
| **紫微斗數 Zi Wei Dou Shu** | Традиционная китайская система астрологии, отображающая судьбу на доску из 12 дворцов, заселённую именованными звёздами. | 五行局 (Класс Пяти Элементов), 命主 (душа) / 身主 (тело), индекс 時辰 (часа) и данные по каждому дворцу — гänzhī, флаги 命/身, диапазон десятилетия и звёзды (основные/вспомогательные/прилагательные с яркостью и 四化). Опционально прогноз 大限/流年 по возможности. |

Тип, Авторитет и Определённость в Human Design **не закодированы** — они выводятся из графика связности определённых центров.

## Почему нативное вычисление

Эта машина выполняет реальные вычисления вместо приближений или вызова внешнего сервиса. Этот выбор даёт три свойства, важные для любого серьёзного инструмента карт:

- **Детерминированность.** Один и тот же вход рождения всегда выдаёт один и тот же выход, побайтово. Нет случайности, нет модели, нет дрейфа округления между запусками.
- **Воспроизводимость.** Любой, у кого есть репозиторий и те же входы, может восстановить вашу точную карту. Вычисления используют Swiss Ephemeris (модель Moshier) для неба и iztro для 紫微斗數 — обе детерминированные.
- **Перекрёстная проверяемость.** Так как три независимые системы вычисляются из одного момента рождения, вы можете триангулировать. **Когда все три системы указывают на один сигнал, считайте это высокой уверенностью. Когда только одна система показывает деталь, считайте её ориентиром, а не выводом.** Это основной принцип дизайна машины — она выдаёт факты для перекрёстного чтения, а не единственный вердикт.

## Быстрый старт

### Установка в одну строку (рекомендуется)

```bash
curl -fsSL https://raw.githubusercontent.com/zhenheco/life-chart-engine/main/install.sh | bash
```

Устанавливается в `~/.life-chart-engine` (можно переопределить через `LIFE_CHART_DIR`). Без `sudo`, без системных изменений — только клонирует репозиторий, создает изолированный venv CPython 3.12 и генерирует закрепленный Node-бандл iztro. Требуются `git`, [`uv`](https://docs.astral.sh/uv/) и Node.js/npm. Повторный запуск в любой момент обновит до последней версии.

### Из исходников

```bash
git clone https://github.com/zhenheco/life-chart-engine.git
cd life-chart-engine
bash setup.sh
```

### Что делает `setup.sh`

Он работает под `set -euo pipefail` и выполняет шесть шагов:

1. **Разрешает расположение виртуального окружения** — `${LIFE_VENV:-<repo>/.venv}`. Установите переменную окружения `LIFE_VENV` для переопределения; по умолчанию `.venv/` игнорируется git.
2. **Предполётная проверка: требует [`uv`](https://docs.astral.sh/uv/)** — если `uv` не в `PATH`, выходит с кодом `1` и выводит подсказку установки:
   ```
   curl -LsSf https://astral.sh/uv/install.sh | sh
   ```
3. **Создаёт виртуальное окружение CPython 3.12** — `uv venv --python 3.12 "$VENV"` (см. [Почему CPython 3.12](#why-cpython-312-specifically)).
4. **Устанавливает зафиксированные зависимости** — `uv pip install --python "$VENV/bin/python" -r requirements.txt`.
5. **Собирает закрепленный бандл iztro** — запускает `scripts/build-iztro-bundle.sh`, который устанавливает `iztro@2.5.8` во временный каталог сборки и создает `vendor/iztro.cjs`.
6. **Запускает smoke-тест** — один раз выполняет движок с фиксированными примерными входными данными (отбрасывая stdout) и при успехе печатает `OK — engine runs.`. Также печатает подсказки использования для обоих режимов.

### Ручные шаги `uv` (без `setup.sh`)

```bash
# 1. Создать виртуальное окружение CPython 3.12 (по умолчанию <repo>/.venv; переопределить через LIFE_VENV)
uv venv --python 3.12 .venv

# 2. Установить зафиксированные зависимости
uv pip install --python .venv/bin/python -r requirements.txt

# 3. Собрать закрепленный бандл iztro
bash scripts/build-iztro-bundle.sh

# 4. (Опционально) smoke-тест
.venv/bin/python scripts/chart_engine.py --json \
  --name "Setup Test" --gender 女 --date 1990-06-15 --time 08:30 \
  --tz 8 --lat 25.0 --lon 121.5 --target 2025-01-01 >/dev/null
```

Прямые зависимости Python закреплены в `requirements.txt`:

```
pyswisseph==2.10.3.2
fastapi==0.128.8
uvicorn==0.39.0
httpx==0.28.1
```

Zi Wei использует сгенерированный Node-бандл, закрепленный `scripts/build-iztro-bundle.sh`:

```
iztro@2.5.8
```

## Почему CPython 3.12 конкретно

Движок сейчас работает на проверенном runtime **CPython 3.12**. Причина одинаково указана в `requirements.txt` и `setup.sh`:

> CPython 3.12 остается закрепленным для этого изменения. Старое ограничение нативной Python-зависимости Zi Wei исчезло, поэтому это можно пересмотреть после проверки оставшихся зависимостей и deployment-образа.

Коротко: 3.12 всё еще является протестированным runtime для этого релиза. Зависимость Zi Wei больше не навязывает эту фиксацию, поэтому повышение версии Python — последующая проверка совместимости, а не часть этого рефакторинга.

## Справочник флагов CLI

Нет флагов с `required=True` — argparse никогда не выдаст ошибку при отсутствии одного. Пропуск `--date`/`--time`/`--tz`/`--lat`/`--lon` молча переходит к встроенному примеру человека (`範例`, рождённый `2000-01-01 12:00`, UTC+8, Taipei 101). Для корректной карты укажите их все.

| Флаг | Тип | Требуется для правильного использования? | По умолчанию | Формат / правило |
|------|-----|---------------------------------------|----|---------|
| `--name` | строка | Нет (косметический) | `"範例"` | Свободный текст; отражается в выводе только. |
| `--gender` | строка | Только для 紫微 | `"女"` | Должна быть ровно `男` или `女` (выбор argparse; что угодно другое → выход `2`). |
| `--date` | строка | **Да** | переход на `2000-01-01` | `YYYY-MM-DD`, разделение на `-`. Нет требования нулевого заполнения. |
| `--time` | строка | **Да** | переход на `12:00` | `HH:MM`, 24-часовой местный часовой формат, разделение на `:`. |
| `--tz` | float | **Да** | переход на `8.0` | Смещение UTC включая летнее время (Тайвань = `8`). Записывается в ключ входа `tz_offset`. |
| `--lat` | float | **Да** | переход на `25.0330` | Широта в десятичных градусах (западные дома/Асц/MC). |
| `--lon` | float | **Да** | переход на `121.5654` | Долгота в десятичных градусах. |
| `--target` | строка | Нет | `"2025-01-01"` | `YYYY-MM-DD`; дата 紫微 справочный период судьбы (運限參考日). |
| `--json` | флаг | Нет | `False` (Markdown) | Наличие → режим JSON; отсутствие → Markdown. Значение не принимает. |

> Машина **не** геокодирует места и не ищет часовые пояса. Вызывающий должен сам преобразовать место → `lat`/`lon`/`tz` — и часовой пояс/летнее время — это наиболее частый источник ошибок, поэтому проверьте смещение UTC, действовавшее на месте рождения и в дату рождения.

## Справочник выводов

Оболочка `--json` имеет семь ключей верхнего уровня в этом порядке:

`ok` · `schema_version` · `input` · `western` · `human_design` · `ziwei` · `meta`

| Блок | Резюме |
|------|--------|
| `ok` | `true` при успехе (`false` в оболочке ошибки). |
| `schema_version` | `"1.0"`. |
| `input` | Эхо нормализованных входов: `name`, `gender`, `date`, `time`, `tz_offset`, `lat`, `lon`, `target` (обратите внимание на `tz_offset`, не `tz`). |
| `western` | Строка `system`, объекты позиции `ascendant`/`midheaven`, массивы `planets[]`, `houses[]` (×12), `aspects[]`. |
| `human_design` | `type`, `authority`, `profile`, `definition`, `incarnation_cross`, `design_date`, массивы `defined_centers[]`, `open_centers[]`, `channels[]`, `gates[]`. |
| `ziwei` | `five_elements_class`, `soul`, `body`, `hour_index`, массив `palaces[]`, `horoscope` (объект или `null`). |
| `meta` | `{ engine, version, ephemeris }` — все литералы (`ephemeris: "Moshier"`). |

Для полного контракта полей — каждого ключа, типа и протокола вызова агентом — см. **[AGENTS.md](./AGENTS.md)**.

### Особенности полей стоит знать

- **`aspects` НЕ ограничены в JSON.** Путь JSON возвращает *каждый* обнаруженный аспект, отсортированные возрастанием по орбу (самые точные первыми). Ограничение в 10 элементов существует только в отчёте Markdown.
- **`ziwei.horoscope` — лучшее приближение и может быть `null`.** Это обёрнуто в `try/except`; при любом исключении сериализуется как `null`. Когда присутствует, это `{ decadal, yearly }`. (Эти подобъекты раскрывают дополнительную внутреннюю структуру — `index`, `mutagen[]`, `stars[][]`, `yearly_dec_star`, и т.д. — сверх документированного заполнителя.)
- **Строки звёзд кодируют яркость + 四化.** Формат `name(brightness)[mutagen]`, каждая часть опциональна — например `紫微(廟)[祿]`, `紫微(廟)`, `天機[祿]` или просто `天機`. `adjective_stars` — только простые имена (без яркости/мутагена).

## Уровни точности

Не каждый выход несёт одинаковую уверенность. Читайте каждый уровень соответственно:

| Уровень | Что покрывает | Уверенность |
|---------|---------------|----|
| **Наивысший** | Долготы планет, знаки, ретроградность, плюс 紫微 размещение звёзд / 命宮·身宮 / 五行局 — чистая эфемеридная и календарная математика. | Астрономически/календарно точно. |
| **Высокий, зависящий от времени** | Асцендент, Середина неба, все 12 куспидов домов, дом, в который попадает каждая планета, линии Human Design и индекс 紫微 時辰. | Точно *при наличии* точного времени рождения; чувствительно к минутам. |
| **Проверенный** | Яркость звёзд 紫微斗數 — согласованная с выводами библиотеки iztro. | Проверено против библиотеки. |
| **Граница флага ±0.3°** | Любая планета / врата / линия сидящие в пределах ±0.3° от границы. | Считайте предварительно и отметьте воздействие — ±0.3° может перевернуть это через линию. |

## Известные ограничения

- **Нет Chiron / малых тел.** Сборка использует эфемериду Moshier (`swe.FLG_MOSEPH`, нет файлов данных), которая не предоставляет Chiron или другие малые тела. Вычисляются только 10 классических планет плюс лунные узлы.
- **紫微斗數 использует одну школу по умолчанию.** iztro вызывается с фиксированными опциями (`bySolar(..., True, 'zh-TW')`); школа размещения звёзд и 四化 — всё, что iztro устанавливает по умолчанию. Если вы обычно используете 飛星 или другую школу, основная структура не меняется, но некоторые детали могут отличаться.
- **Приблизительное время рождения снижает уровень зависимости от времени.** Если время рождения неточно, назначения Асцендента/MC/домов, линии Human Design и 時辰 — и всё, что от них зависит — становятся недежурными. В этом случае **считайте поля, зависящие от времени, предварительными** и рассмотрите **событийную ректификацию** (согласование известных событий жизни с временем карты) перед использованием.

Предусмотренная модель интеграции — **самостоятельная установка**, не SaaS.

Пользователь копирует URL этого репозитория и **свой собственный** агент или CLI (Claude Code, Hermes, скрипт и т.д.) клонирует его и запускает **локально на машине пользователя**. Вычисление происходит на стороне пользователя. Нет размещённой точки входа для вызова, нет учётной записи и **не требуется интеграция SaaS** — машина — это эгалитарный, детерминированный, автономный подпроцесс.

Так как издатель не эксплуатирует его как сетевой сервис, на издателя не падает обязательство §13 для чьей-либо самостоятельной установки. (Отдельно, §13 только обязует оператора, который запускает *модифицированную* сетьевую версию — не немодифицированное локальное использование.)

Для агентов контракт прост: dispatch подпроцесса `--json` с Python из виртуального окружения репозитория в рабочем каталоге, разберите stdout как JSON, ветвитесь на `ok` (и код выхода), затем передайте структурированный объект. Очистка не требуется — это эгалитарно. Полный контракт CLI + JSON находится в **[AGENTS.md](./AGENTS.md)**.

## FAQ

**Можно ли ввести лунную дату?**
Нет. Входы — это григорианская солнечная дата (`--date YYYY-MM-DD`) и часовой формат (`--time HH:MM`). Если у вас есть только лунная дата, сначала преобразуйте её. 紫微斗數 вычисляется через точку входа солнца iztro (`bySolar`).

**Моё время рождения только приблизительно — это проблема?**
Позиции планет в порядке, но Асцендент, Середина неба, куспиды домов, дом, в котором находится каждая планета, линии Human Design и 時辰 — все чувствительны ко времени. Считайте их предварительными и рассмотрите событийную ректификацию перед использованием.

**Где Chiron / астероиды / малые тела?**
Не включены. Эфемерида Moshier, используемая здесь, их не предоставляет; вычисляются только 10 классических планет и лунные узлы.

**Какую школу 紫微斗數 она использует?**
Школу по умолчанию как реализовано iztro (`bySolar(..., True, 'zh-TW')`). Школа не выбирается пользователем.

**Звонит ли домой / использует сеть?**
Нет. Машина полностью автономна — нет телеметрии, нет сетевых вызовов, нет побочных эффектов. Это эгалитарный, детерминированный одноразовый подпроцесс.

**Могу ли я использовать её в закрытом продукте?**
Под AGPL-3.0 — да для приватного/локального использования. Распространение сборки вызывает обязательства GPL conveying/исходника, и сетевая обработка *модифицированной* сборки вызывает AGPL §13 — в любом случае вы должны предложить соответствующий исходник. Для полной закрытости купите коммерческую лицензию Swiss Ephemeris у Астродиенста или поменяйте эфемериду на Skyfield + JPL DE440 (см. [Лицензирование](#licensing)).
