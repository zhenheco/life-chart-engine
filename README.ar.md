# life-chart-engine

**حساب حتمي محلي لثلاث أنظمة رسوم حياة — علم التنجيم الغربي للميلاد، 人類圖 (Human Design)، و紫微斗數 (Zi Wei Dou Shu) — من سجل ميلاد واحد فقط.**

[![License: AGPL-3.0](https://img.shields.io/badge/License-AGPL--3.0-blue.svg)](./LICENSE)
[![Python 3.12](https://img.shields.io/badge/Python-3.12-3776AB.svg)](#لماذا-cpython-312-تحديدا)
[![No telemetry · offline](https://img.shields.io/badge/no%20telemetry-offline-green.svg)](#الأسئلة-الشائعة)

`life-chart-engine` أداة سطر أوامر صغيرة وعاملة بدون اتصال. تزودها بيانات ميلاد شخص واحد — التاريخ والوقت والمنطقة الزمنية وإحداثيات المكان — وتقوم بحساب ثلاثة أنظمة مستقلة في مسار واحد، ثم تصدر تقرير Markdown قابل للقراءة أو كائن JSON منظم لتستهلكه البرامج والعملاء الذكيين.

مصممة للأشخاص الذين يريدون حسابات رسوم **قابلة للتكرار والتحقق** بدلاً من تطبيق ويب صندوق أسود: الممارسون والمطورون الذين يبنون أدوات الوعي الذاتي، والعملاء الذكيين الذين يحتاجون إلى خطوة حساب نقية. كل رقم يأتي من حسابات فلكية حقيقية (Swiss Ephemeris) ومكتبة 紫微斗數 حقيقية (iztro) — ليس من خدمة بعيدة، وليس من بحث مخزن مؤقتاً، وليس أبداً عبر الشبكة.

## ما الذي يتم حسابه

يشغل المحرك ثلاثة أنظمة ضد **نفس لحظة الميلاد**، بحيث يمكن لمخرجاتها أن تؤكد بعضها البعض. كل نظام يجيب على نوع مختلف من الأسئلة:

| النظام | ما هو (بلغة عادية) | ما يعطيه المحرك |
|--------|--------------------------|------------------------|
| **Western natal** (Tropical / Placidus) | علم التنجيم الغربي الكلاسيكي — حيث جلست الكواكب مقابل دائرة البروج عند ولادتك، مقسومة إلى 12 بيت. | الطالع + كلف السماء، 12 كوكب/نقطة (太陽 → 南交點) مع البرج والدرجة والبيت وعلم الرجعية، جميع 12 حافة البيوت، وكل جانب مكتشف (合相/六分/四分/三分/對分) مرتب حسب الضيق. |
| **人類圖 Human Design** | تركيب حديث من علم التنجيم ومدونة أيتشينج ونظام تشاكرا. يصف كيفية "توصيل" طاقتك عبر البوابات والقنوات والمراكز. | النوع والسلطة والملف الشخصي والتعريف وصليب التجسد والتاريخ التصميمي بـ 88° سابقاً والمراكز المعرفة/المفتوحة والقنوات المعرفة وتنشيطات الخط البوابة لكل كوكب لكل من رسوم الشخصية والتصميم. |
| **紫微斗數 Zi Wei Dou Shu** | نظام التنجيم الصيني التقليدي الذي يرسم الحظ على لوحة بـ 12 قصر، مأهول بنجوم مسماة. | 五行局 (فئة العناصر الخمسة)، 命主 (الروح) / 身主 (الجسد)، مؤشر الساعة 時辰، وبيانات لكل قصر — ganzhi، علامات 命/身، نطاق السن العقدي، والنجوم الرئيسية/الثانوية/الصفة (مع السطوع و四化). اختياراً تنبؤ 大限/流年 أفضل جهد. |

النوع والسلطة والتعريف في Human Design **لا يتم ترميزها بشكل ثابت** — يتم اشتقاقها من رسم البياني للاتصال للمراكز المعرفة.

## لماذا الحسابات المحلية

يفعل هذا المحرك الرياضيات الحقيقية بدلاً من التقريب أو استدعاء خدمة. هذا الاختيار يشتري ثلاث خصائص مهمة لأي أداة رسم جادة:

- **حتمي.** نفس المدخل الميلادي ينتج دائماً نفس المخرجات، بايت مقابل بايت. لا يوجد عشوائية، لا نموذج، لا انجراف التقريب بين التشغيل.
- **قابل للتكرار.** يمكن لأي شخص لديه المستودع والمدخلات نفسها إعادة توليد رسمك الدقيق. الحسابات تستخدم Swiss Ephemeris (نموذج Moshier) للسماء و iztro ل 紫微斗數 — كلاهما حتمي.
- **قابل للتحقق المتبادل.** لأن ثلاثة أنظمة مستقلة يتم حسابها من لحظة ميلاد واحدة، يمكنك الثلاثة. **عندما تشير جميع الأنظمة الثلاثة إلى نفس الإشارة، عاملها كثقة عالية. عندما يظهر نظام واحد فقط التفصيل، عامله كنقطة مرجعية، ليس كنتيجة.** هذا هو مبدأ التصميم الأساسي للمحرك — ينتج حقائق للقراءة المتقاطعة، وليس حكماً واحداً.

## البدء السريع

### تثبيت بسطر واحد (موصى به)

```bash
curl -fsSL https://raw.githubusercontent.com/zhenheco/life-chart-engine/main/install.sh | bash
```

يُثبَّت في `~/.life-chart-engine` (يمكن التجاوز بـ `LIFE_CHART_DIR`). لا `sudo` ولا تغييرات على مستوى النظام — ينسخ المستودع فقط، ويبني venv معزولًا لـ CPython 3.12، وينشئ حزمة iztro Node المثبتة. يتطلب `git` و[`uv`](https://docs.astral.sh/uv/) و Node.js/npm. أعد تشغيله في أي وقت للتحديث إلى أحدث إصدار.

### من المصدر

```bash
git clone https://github.com/zhenheco/life-chart-engine.git
cd life-chart-engine
bash setup.sh
```

### ما يفعله `setup.sh`

يعمل تحت `set -euo pipefail` وينفذ ست خطوات:

1. **يحل موقع venv** — `${LIFE_VENV:-<repo>/.venv}`. عيّن متغير البيئة `LIFE_VENV` للتجاوز؛ افتراضي `.venv/` هو git-ignored.
2. **الفحص المسبق: يتطلب [`uv`](https://docs.astral.sh/uv/)** — إذا لم يكن `uv` على `PATH` ينخفض `1` ويطبع تلميح التثبيت:
   ```
   curl -LsSf https://astral.sh/uv/install.sh | sh
   ```
3. **ينشئ venv CPython 3.12** — `uv venv --python 3.12 "$VENV"` (انظر [لماذا CPython 3.12](#لماذا-cpython-312-تحديدا)).
4. **يثبت المتطلبات المثبتة** — `uv pip install --python "$VENV/bin/python" -r requirements.txt`.
5. **يبني حزمة iztro المثبتة** — يشغل `scripts/build-iztro-bundle.sh`، الذي يثبت `iztro@2.5.8` في دليل بناء مؤقت وينتج `vendor/iztro.cjs`.
6. **يشغل اختبار دخان** — ينفذ المحرك مرة واحدة بمدخلات عينة ثابتة (مع تجاهل stdout) ويطبع `OK — engine runs.` عند النجاح. كما يطبع تلميحات الاستخدام لكلا الوضعين.

### خطوات `uv` اليدوية (بدون `setup.sh`)

```bash
# 1. إنشاء venv CPython 3.12 (افتراضي <repo>/.venv؛ تجاوز مع LIFE_VENV)
uv venv --python 3.12 .venv

# 2. تثبيت pinned deps
uv pip install --python .venv/bin/python -r requirements.txt

# 3. بناء حزمة iztro المثبتة
bash scripts/build-iztro-bundle.sh

# 4. (اختياري) اختبار دخان
.venv/bin/python scripts/chart_engine.py --json \
  --name "Setup Test" --gender 女 --date 1990-06-15 --time 08:30 \
  --tz 8 --lat 25.0 --lon 121.5 --target 2025-01-01 >/dev/null
```

اعتماديات Python المباشرة مثبتة في `requirements.txt`:

```
pyswisseph==2.10.3.2
fastapi==0.128.8
uvicorn==0.39.0
httpx==0.28.1
```

تستخدم 紫微斗數 حزمة Node مُنشأة ومثبتة بواسطة `scripts/build-iztro-bundle.sh`:

```
iztro@2.5.8
```

> يبقى CPython 3.12 مقفلًا لهذا التغيير. انتهى قيد اعتماد Zi Wei الأصلي في Python، لذلك يمكن إعادة النظر في ذلك بعد فحص بقية الاعتماديات وصورة النشر.

## مرجع أعلام سطر الأوامر

**لا توجد أعلام `required=True`** — argparse لا تخطئ على واحد مفقود. ترك `--date`/`--time`/`--tz`/`--lat`/`--lon` بدون ملء بصمت يسقط إلى شخص مثال مدمج (`範例`، ولد `2000-01-01 12:00`، UTC+8، Taipei 101). لذا للحصول على رسم صحيح، زود كل ذلك.

| العلم | النوع | مطلوب للاستخدام الصحيح؟ | الافتراضي | الصيغة / القاعدة |
|------|------|---------------------------|---------|---------------|
| `--name` | string | لا (تجميلي) | `"範例"` | نص حر؛ متردد في المخرجات فقط. |
| `--gender` | string | فقط ل 紫微 | `"女"` | يجب أن يكون `男` أو `女` بالضبط (argparse `choices`؛ أي شيء آخر → خروج `2`). |
| `--date` | string | **نعم** | يسقط إلى `2000-01-01` | `YYYY-MM-DD`، قسم على `-`. بدون متطلب الحشو الصفر. |
| `--time` | string | **نعم** | يسقط إلى `12:00` | `HH:MM`، ساعة 24 محلية، قسم على `:`. |
| `--tz` | float | **نعم** | يسقط إلى `8.0` | إزاحة UTC بما في ذلك DST (تايوان = `8`). مكتوب إلى مفتاح الإدخال `tz_offset`. |
| `--lat` | float | **نعم** | يسقط إلى `25.0330` | خط العرض بالدرجات العشرية (البيوت الغربية/Asc/MC). |
| `--lon` | float | **نعم** | يسقط إلى `121.5654` | خط الطول بالدرجات العشرية. |
| `--target` | string | لا | `"2025-01-01"` | `YYYY-MM-DD`؛ تاريخ مرجعي فترة الحظ 紫微 (運限參考日). |
| `--json` | flag | لا | `False` (Markdown) | الوجود → وضع JSON؛ الغياب → Markdown. لا يأخذ قيمة. |

> المحرك **لا يقوم بـ** تحويل الأماكن الجغرافية أو البحث عن المناطق الزمنية. يجب على المتصل تحويل المكان → `lat`/`lon`/`tz` بنفسه — والمنطقة الزمنية/DST هي المصدر الأكثر شيوعاً للخطأ، لذا تحقق من إزاحة UTC التي كانت سارية في مكان الميلاد وتاريخ الميلاد.

## مرجع المخرجات

غلاف `--json` له سبعة مفاتيح على مستوى أعلى، بهذا الترتيب:

`ok` · `schema_version` · `input` · `western` · `human_design` · `ziwei` · `meta`

| الكتلة | الملخص |
|-------|---------|
| `ok` | `true` عند النجاح (`false` في غلاف الخطأ). |
| `schema_version` | `"1.0"`. |
| `input` | صدى المدخلات المعايرة: `name`، `gender`، `date`، `time`، `tz_offset`، `lat`، `lon`، `target` (لاحظ `tz_offset`، وليس `tz`). |
| `western` | سلسلة `system`، كائنات موضع `ascendant`/`midheaven`، `planets[]`، `houses[]` (×12)، `aspects[]`. |
| `human_design` | `type`، `authority`، `profile`، `definition`، `incarnation_cross`، `design_date`، `defined_centers[]`، `open_centers[]`، `channels[]`، `gates[]`. |
| `ziwei` | `five_elements_class`، `soul`، `body`، `hour_index`، `palaces[]`، `horoscope` (كائن أو `null`). |
| `meta` | `{ engine, version, ephemeris }` — كل الحرفيات (`ephemeris: "Moshier"`). |

للعقد الكامل — كل مفتاح وكل نوع والبروتوكول استدعاء الوكيل — انظر **[AGENTS.md](./AGENTS.md)**.

### الحقول الطريفة الجديرة بالمعرفة

- **`aspects` لا تحدها في JSON.** مسار JSON يعيد *كل* جانب مكتشف، مرتب صعوداً حسب orb (أضيق أولاً). حد 10 عناصر موجود فقط في تقرير Markdown.
- **`ziwei.horoscope` أفضل جهد وقد تكون `null`.** تُلفقت في `try/except`؛ على أي استثناء تسلسل كـ `null`. عندما تكون موجودة هي `{ decadal, yearly }`. (هذه الكائنات الفرعية تعرض هيكل داخلي إضافي — `index`، `mutagen[]`، `stars[][]`، `yearly_dec_star`، إلخ. — بعيداً عن العنصر النائب الموثق.)
- **سلاسل النجوم ترميز السطوع + 四化.** الصيغة هي `name(brightness)[mutagen]`، مع كل جزء اختياري — مثل `紫微(廟)[祿]`، `紫微(廟)`، `天機[祿]`، أو عادي `天機`. `adjective_stars` هي أسماء عادية فقط (بدون سطوع/mutagen).

## القيود المعروفة

- **لا كايرون / الأجسام الصغيرة.** البناء يستخدم إفيميريس Moshier (`swe.FLG_MOSEPH`، بدون ملفات بيانات)، التي لا توفر كايرون أو أجسام صغيرة أخرى. فقط 10 كواكب كلاسيكية بالإضافة إلى العقد القمرية تُحسب.
- **紫微斗數 تستخدم مدرسة افتراضية واحدة.** يتم استدعاء iztro بخيارات ثابتة (`bySolar(..., True, 'zh-TW')`؛ مدرسة الوضع النجمي و四化 هي كل ما iztro يأتي افتراضياً إليه. إذا كنت تستخدم عادة 飛星 أو مدرسة أخرى، البنية الرئيسية لم تتغير لكن بعض التفاصيل قد تختلف.
- **وقت الميلاد التقريبي يتدهور طبقة المعتمد على الوقت.** إذا كان وقت الميلاد غير مؤكد، الطالع/MC/تعيينات البيت، خطوط Human Design، و時辰 — وأي شيء مشتق منها — تصبح غير موثوقة. في تلك الحالة، **تعامل مع حقول الوقت المعتمد كمؤقتة** واعتبر **تصحيح بناءً على الحدث** (مطابقة الأحداث الحياتية المعروفة مع توقيت الرسم) قبل الاعتماد عليها.

## استخدمها من عميل ذكي / Hermes

نموذج التكامل المقصود هو **self-install**، وليس SaaS.

يقوم المستخدم بنسخ رابط هذا المستودع، و**نسخته الخاصة** من عميل أو CLI (Claude Code، Hermes، سكريبت، إلخ.) بنسخه وتشغيله **محلياً على جهاز المستخدم**. الحسابات تحدث على جانب المستخدم. لا توجد نقطة نهاية مستضافة للاتصال بها، لا حساب، و**لا تكامل SaaS مطلوب** — المحرك هو عملية عديمة الحالة وحتمية وعاملة بدون اتصال.

لأن الناشر لا يشغله كخدمة شبكة، لا التزام §13 يسقط على الناشر لـ self-install شخص آخر. (بشكل منفصل، §13 يلزم فقط مشغل يشغل نسخة *معدلة* يمكن الوصول إليها عبر الشبكة — وليس استخدام محلي غير معدل.)

بالنسبة للعملاء، العقد بسيط: اسحب العملية الفرعية `--json` مع venv Python في workdir المستودع، حلل stdout كـ JSON، فرع على `ok` (وكود الخروج)، ثم سلم الكائن المنظم. لا تنظيف مطلوب — إنه عديم الحالة. العقد CLI + JSON الكامل يسكن في **[AGENTS.md](./AGENTS.md)**.

### إعداد نسخ واللصق لوكيلك الذكي

الصق هذا الكتلة مباشرة في Claude Code أو ChatGPT أو أي عميل ترميز — يخبر العميل كيفية التثبيت والاستدعاء:


## الأسئلة الشائعة

**هل يمكنني إدخال تاريخ قمري؟**
لا. المدخلات هي تاريخ جريجوري شمسي (`--date YYYY-MM-DD`) ووقت ساعة (`--time HH:MM`). إذا كان لديك فقط تاريخ قمري، قم بتحويله أولاً. 紫微斗數 يتم حسابه عبر نقطة الدخول الشمسية iztro (`bySolar`).

**وقت ميلادي تقريبي فقط — هل هذه مشكلة؟**
مواضع الكواكب بخير، لكن الطالع والكلف وحافات البيوت والبيت الذي يجلس فيه كل كوكب وخطوط Human Design و時辰 كلها حساسة للوقت. تعامل معها كمؤقتة واعتبر تصحيح قائم على الحدث قبل الاعتماد عليها.

**أين كايرون / الكويكبات / الأجسام الصغيرة؟**
غير مضمونة. إفيميريس Moshier المستخدمة هنا لا توفرها؛ فقط 10 كواكب كلاسيكية والعقد القمرية تُحسب.

**أي مدرسة 紫微斗數 تستخدم؟**
مدرسة الافتراضية كما طبقت iztro (`bySolar(..., True, 'zh-TW')`). المدرسة ليست قابلة لاختيار المستخدم.

**هل تتصل بالمنزل / استخدام الشبكة؟**
لا. المحرك محلي بالكامل — لا تلميح جمع البيانات، لا استدعاءات شبكة، لا آثار جانبية. إنه عملية فرعية حتمية عديمة الحالة لمرة واحدة.

**هل يمكنني استخدامها داخل منتج مغلق المصدر؟**
تحت AGPL-3.0، نعم للاستخدام الخاص/المحلي. توزيع بناء يؤدي إلى التزامات GPL نقل/المصدر، وخدمة الشبكة لـ *تعديل* بناء تؤدي إلى AGPL §13 — على أي حال يجب عليك تقديم المصدر المقابل. للذهاب المغلق بالكامل، إما اشتر ترخيص Swiss Ephemeris تجاري من Astrodienst أو استبدل الإفيميريس بـ Skyfield + JPL DE440 (انظر [الترخيص](#الترخيص)).
